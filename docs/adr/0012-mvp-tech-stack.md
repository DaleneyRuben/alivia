# MVP tech stack: Next.js, Postgres, Prisma, Vitest

The app needs one codebase serving both the patient-facing booking flow and the Doctor/Assistant panel, built by a solo/small team with no dedicated ops. We're using Next.js (App Router, TypeScript) for both, Tailwind CSS for styling (bespoke components, no UI kit), Postgres via Prisma, Vitest + Testing Library for tests, and Yarn as the package manager.

TypeScript keeps the whole app in one language matching the Airbnb JS style guide already mandated project-wide, rather than splitting frontend/backend across ecosystems — a Rails monolith and a separate SPA + API were both considered and rejected for that reason. Postgres fits the domain model's relational structure (Doctor → Location → Schedule → Slot → Appointment). Prisma was chosen over Drizzle and Sequelize because its migrations produce a reviewable SQL diff, matching the project rule that no migration runs without the user approving the proposed schema change first — Sequelize's hand-written migration files don't give that same enforced diff, and its TypeScript support isn't generated from the schema the way Prisma's and Drizzle's are. Vitest was chosen over Jest for faster feedback in the Red → Green → Refactor loop the TDD workflow requires for every change.

Mutations (booking an Appointment, updating a Schedule, marking Attended/No-show) use Next.js Server Actions rather than a separate API layer plus a client-side data library — most of this app's flows are submit-and-confirm rather than live-updating, and Server Actions are the idiomatic App Router pattern with less to learn on top of a framework the team is new to. TanStack Query is deferred until a specific screen (e.g. the Assistant panel's day-before Confirmation list) needs client-side caching or optimistic updates; it can be layered on top of the same Server Actions later without rework.

Auth.js (NextAuth) handles Doctor/Assistant login — the only accounts in the system, since Patients book as guests (see [0006](0006-guest-booking-no-patient-accounts.md)) — rather than a hand-rolled session/password implementation or a paid hosted provider (Clerk/Auth0), which is more than concierge-only onboarding (see [0005](0005-concierge-doctor-onboarding.md)) currently needs. React Hook Form + Zod handles form state and validation (Onboarding, booking, Schedule blocks), with the same Zod schema reused for server-side validation inside Server Actions. date-fns handles the Slot-generation and availability-sorting math (see [0008](0008-availability-first-search-no-location-filter.md)); since the MVP is La Paz-only with a single, unchanging UTC-4 offset (see [0001](0001-mvp-scope-doctors-la-paz-only.md)), the heavier multi-timezone machinery of Luxon isn't needed. The Assistant panel's tables, dialogs, and date pickers are built bespoke with Tailwind rather than a component kit like shadcn/ui, keeping every visual element under direct control rather than adopting someone else's design system.

Deployment is Vercel (built by the Next.js team, zero-config git-push deploys) paired with Neon for managed Postgres — chosen over Supabase because Supabase's bundled auth layer would go unused now that Auth.js is the auth choice, and over Vercel Postgres to keep the database on its own dashboard/billing rather than fully inside Vercel's.
